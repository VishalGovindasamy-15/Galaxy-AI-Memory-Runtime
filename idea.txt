### Traditional game loading

Most older games work like this:

```
SSD/HDD
   ↓
Load a large chunk into RAM
   ↓
CPU decompresses
   ↓
GPU receives textures/models
   ↓

```

The game loads lots of assets into RAM because disk access was relatively slow. As you move around, it unloads old assets and loads new ones.

This has several problems:

* High RAM usage
* Long loading screens
* CPU becomes a bottleneck
* Objects may "pop in"

Modern engines instead treat storage almost like an extension of memory.

```
SSD
```

## Research challenges

Several difficult problems would need to be solved:

* **Prediction accuracy.** If the scheduler predicts the wrong weights, the GPU stalls waiting for data.
* **Latency.** Token generation is sequential, leaving limited time to fetch weights.
* **Bandwidth.** Even fast PCIe and NVMe are much slower than VRAM bandwidth.
* **Training state.** Optimizers like Adam require extra memory for gradients and momentum, making full training much harder than inference.

## One thing to be careful about

The biggest bottleneck isn't always the order of the layers—it's **bandwidth**.

For example:

* GPU VRAM bandwidth: ~300–1000+ GB/s
* PCIe Gen4 x16: ~32 GB/s
* Fast NVMe SSD: ~7–14 GB/s

Even if your scheduling is perfect, if a layer is larger than what can be transferred while the previous layer is computing, the GPU will still have to wait.

So the key research question becomes:

> **Can we overlap I/O, decompression, and computation well enough that data transfer is effectively hidden behind computation?**

If the answer is yes—even for a significant fraction of layers—you could run much larger models on consumer GPUs with much better performance.

A transformer layer contains:

where (d_{model}) is the hidden dimension (the layer width).

Notice the **square**.

If you double the width:


